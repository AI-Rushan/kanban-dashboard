- **Напоминание о файлах:** после каждого изменения кода явно напоминать скопировать обновлённый файл из рабочей папки в `~/Desktop/kanban/`

## Архитектурные решения

### Общее
- **Один HTML-файл** — никаких фреймворков, сборщиков, зависимостей. Всё в `dashboard.html` и `archive.html`
- **Хранение данных** — `localStorage` браузера, без серверной части
- **Запуск** — через `launch.command` (локальный Python-сервер на порту 8080) для поддержки Google Calendar OAuth; без Calendar достаточно двойного клика на HTML-файле

### Архив
- Задачи со статусом «Сделано» автоматически переходят в архив **после завершения текущей ISO-недели** (задачи выполненные на текущей неделе остаются видимыми на доске)
- Архив — отдельная страница `archive.html`, не колонка на доске
- Архивные задачи показываются сгруппированными по направлениям, с возможностью сворачивания групп

### Направления (Areas)
- Направления динамические — можно добавлять, редактировать, удалять через интерфейс
- Цвет направления задаётся через палитру из 12 цветов
- У каждого направления: emoji, название, цвет рамки/фона/текста карточки

## Google Calendar — согласованные решения

### Общая логика
- **Не создаём отдельный календарь** «Задачи» — используем существующий выбранный пользователем
- Связь задача ↔ событие через скрытое поле `extendedProperties.private.taskId`
- **Синхронизация запускается автоматически** при каждом открытии дашборда (если Calendar подключён)
- Ручной запуск — кнопка 🔄 в тулбаре

### Направление: Дашборд → Calendar
- Задача с дедлайном + время → timed event (длительность 1 час)
- Задача с дедлайном без времени → all-day event
- Задача «Сделано» → дата события = `completedAt`
- Задача «Архив» → событие остаётся в Calendar, цвет становится серым (не удаляется)
- Задача без дедлайна → событие не создаётся
- Статус задачи определяет цвет события: backlog=серый, todo=синий, doing=оранжевый, done=зелёный, archive=серый

### Направление: Calendar → Дашборд
- **Все события** из выбранного календаря попадают в дашборд (не требуется тег #задача)
- Принятые приглашения (`responseStatus: 'accepted'`) — синхронизируются
- Непринятые и отклонённые — не синхронизируются
- События без признака принадлежности (нет attendees) — считаются своими, синхронизируются
- Окно синхронизации: начало текущей недели + 32 дня вперёд (для охвата повторяющихся событий)
- `singleEvents=true` — повторяющиеся события разворачиваются в отдельные вхождения

### Повторяющиеся события — Вариант А (согласован)
- Каждое вхождение повторяющегося события создаёт отдельную задачу
- Прошедшие вхождения остаются как задачи в статусе backlog (просроченные)
- Пользователь сам решает что делать с просроченной задачей

### Разрешение конфликтов
- **Last-write-wins** по полю `lastModified` (timestamp)
- Если событие в Calendar обновлено позже чем задача → применяем данные из Calendar
- Если задача обновлена позже → пушим данные в Calendar

### Ремайндеры в Calendar
- Ремайндеры не влияют на синхронизацию и не смещают дедлайны
- Ремайндеры — отдельные метаданные события, не затрагиваются

## UX-решения

- Уведомления о дедлайнах — браузерные push-уведомления, проверка раз в час
- Описание задачи — свободное поле, ссылки автоматически становятся кликабельными
- Время дедлайна — необязательное поле, обратно совместимо (старые задачи без времени работают)
- Фильтрация — по направлениям через чипы в тулбаре
- Сортировка — независимая для каждой колонки, по дедлайну / направлению / дате завершения

## Экспорт / Импорт

- Формат: JSON с полями `version`, `app`, `exportedAt`, `tasks`, `areas`
- Экспорт — кнопка 📤 в тулбаре, скачивает файл `kanban_backup_ГГГГ-ММ-ДД.json`
- Импорт — кнопка 📥 (label с hidden input), данные применяются без перезагрузки страницы
- При импорте выполняется миграция: добавляются отсутствующие поля (`gcalEventId`, `lastModified`, `deadlineTime`, `description`)
