# Требования к продукту «Канбан-дашборд»

Версия: 1.0  
Дата: 2026-06-11  
Владелец продукта: Рушан Умеров

---

## 1. Назначение продукта

Персональный канбан-дашборд для управления личными задачами. Целевые пользователи: владелец (IT-руководитель, AI-консультант) и его дочери. Продукт работает как локальное веб-приложение без постоянного подключения к интернету.

---

## 2. Функциональные требования

### 2.1 Канбан-доска

- Четыре колонки с фиксированными статусами:
  - **Поток задач (Backlog)** — всё придуманное, входящие задачи
  - **Спринт (Todo)** — задачи, взятые в работу на этой неделе
  - **В работе (Doing)** — активно выполняемые прямо сейчас
  - **Сделано (Done)** — завершённые задачи
- Счётчик задач в заголовке каждой колонки
- Статистика в шапке страницы: количество задач по каждому статусу + количество просроченных

### 2.2 Задачи

Каждая задача содержит:
- **Текст** — описание задачи (обязательное)
- **Направление** — к какой сфере жизни относится (обязательное)
- **Дедлайн** — дата (необязательное)
- **Время дедлайна** — время в формате HH:MM (необязательное, отдельно от даты)
- **Описание** — свободное поле для заметок, ссылок, повестки встречи и т.д. (необязательное)
- **Статус** — одна из четырёх колонок

Поведение задач:
- Перемещение между колонками: перетаскиванием (drag & drop) или кнопками ← →
- При переходе в «Сделано» автоматически фиксируется дата завершения
- Визуальная индикация статуса дедлайна: просрочено / сегодня / скоро (≤3 дней) / в норме
- Ссылки в поле описания автоматически становятся кликабельными

### 2.3 Направления (Areas)

- Минимум 7 предустановленных направлений (с возможностью изменения)
- Каждое направление: emoji, название, цвет
- Операции: добавить новое, редактировать существующее, удалить (с предупреждением если есть задачи)
- Цвет выбирается из палитры из 12 вариантов
- Фильтрация задач на доске по направлению — через чипы в тулбаре

### 2.4 Сортировка

- Независимая сортировка для каждой колонки
- Варианты сортировки:
  - По дедлайну (по возрастанию / убыванию)
  - По направлению
  - По дате завершения (только для колонки «Сделано»)
- Иконки сортировки в заголовке каждой колонки

### 2.5 Архив

- Задачи со статусом «Сделано», завершённые до текущей ISO-недели, автоматически переходят в архив
- Задачи, завершённые на текущей неделе, остаются видимыми на доске
- Архив — отдельная страница `archive.html`
- На странице архива:
  - Группировка по направлениям (сворачиваемые секции)
  - Фильтрация по направлениям
  - Сортировка: по дате завершения, по дате дедлайна
  - Статистика: всего задач / выполнено вовремя / с опозданием
  - Бейдж для каждой задачи: вовремя / с опозданием / без дедлайна
  - Отображение описания (раскрывающийся блок)

### 2.6 Уведомления

- Браузерные push-уведомления о дедлайнах
- Уведомление при статусе «просрочено» или «сегодня»
- Проверка не чаще раза в час (чтобы не спамить)
- Кнопка 🔔 в тулбаре для разрешения/проверки уведомлений

### 2.7 Google Calendar — двусторонняя синхронизация

**Из дашборда в Calendar:**
- Задача с дедлайном → событие в Google Calendar
- Задача с временем дедлайна → timed-событие (1 час)
- Задача без времени → all-day событие
- Статус задачи отображается цветом события
- Архивные задачи остаются в Calendar (серый цвет), не удаляются
- Поле описания задачи → описание события

**Из Calendar в дашборд:**
- Все события из выбранного пользователем календаря → задачи в Backlog
- Принятые приглашения синхронизируются, отклонённые — нет
- Повторяющиеся события разворачиваются в отдельные задачи (32-дневное окно)
- Прошедшие вхождения остаются как задачи (просроченные)

**Конфликты:** побеждает более свежая запись (по timestamp `lastModified`)

**Настройка:**
- Ввод Google Cloud OAuth Client ID
- Выбор конкретного календаря из списка доступных
- Автосинхронизация при открытии страницы
- Ручная синхронизация через кнопку 🔄

### 2.8 Экспорт / Импорт данных

- Экспорт всех задач и направлений в JSON-файл (кнопка 📤)
- Импорт из JSON-файла с заменой текущих данных (кнопка 📥)
- Формат файла: `kanban_backup_ГГГГ-ММ-ДД.json`

## 3. Нефункциональные требования

### 3.1 Производительность
- Страница загружается мгновенно (нет внешних зависимостей кроме скрипта GIS для Calendar)
- Рендеринг доски при любом количестве задач без заметных задержек

### 3.2 Надёжность данных
- Все данные сохраняются в localStorage после каждого действия
- Данные не теряются при закрытии/открытии браузера
- Резервное копирование через экспорт в JSON

### 3.3 Простота использования
- Двойной клик на `launch.command` — единственное действие для запуска
- Для базового использования (без Calendar) достаточно двойного клика на `dashboard.html`
- Интерфейс на русском языке

### 3.4 Переносимость
- Вся логика в двух HTML-файлах + один bash-скрипт
- Перенос на другой компьютер: скопировать папку + импортировать JSON-бэкап
- Работает в Safari и Chrome на macOS

### 3.5 Безопасность
- OAuth-токен Google хранится только в памяти (не в localStorage)
- Client ID хранится в localStorage (это нормально — он публичный)
- Данные не передаются на сторонние серверы (только в Google Calendar по запросу пользователя)

---

## 4. Ограничения и допущения

- Только macOS (скрипт `launch.command` — bash)
- Только один пользователь на устройство (localStorage не разделяет пользователей)
- Google Calendar OAuth работает только через `http://localhost` (не через `file://`)
- Токен Google живёт ~1 час, после чего нужна повторная авторизация при следующей синхронизации
- Приложение находится в режиме «Тестирование» Google OAuth — нужно добавлять пользователей вручную в Cloud Console
