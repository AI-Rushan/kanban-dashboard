## Сессия 7 — Тайм-зоны, тестирование, BUG-01 (2026-06-14)

### Что делали

**Обнаружили баг с датой завершения** — задача «RedMadRobots-AI_SDLC» перенесена в «Сделано» 14 июня, но `completedAt` сохранилось как `2026-06-13`. Причина: `todayStr()` использовал `.toISOString().slice(0,10)` → UTC-дата, которая для UTC+3 до 03:00 ночи = вчера.

**Провели аудит UTC** — нашли 4 места где дата бралась из UTC вместо локального времени:
1. `todayStr()` — дата completedAt при переводе в Done
2. `gcalAddDay()` — `end.date` всесуточных событий Calendar
3. Имя файла при экспорте (`kanban_backup_ДАТА.json`)
4. `gcalLocalISO()` + `timeMin`/`timeMax` в `syncFull()`

Все 4 исправления применили — теперь все даты берутся через `getFullYear()` / `getMonth()` / `getDate()` (локальные компоненты).

**Написали автотесты** — два файла:
- `test_dates.js` (13 тестов) — чистые функции работы с датами, мок Date на 01:30 МСК
- `run_tests.js` (37 тестов, 10 ⏭) — извлекает весь JS из dashboard.html и тестирует: даты, синхронизацию, разрешение конфликтов, архивирование. Запуск: `TZ=Europe/Moscow node run_tests.js`

**Обнаружили BUG-01** — тест `dateStatus-today` упал: функция возвращала `'soon'` для всесуточной задачи с дедлайном сегодня.
- Причина: `deadlineDt` возвращает `23:59:59`, `TODAY` = `00:00:00`, разница ≈ 24ч → `Math.ceil(0.9999) = 1` → `'soon'`
- Условие `diffDays === 0` для всесуточных событий недостижимо
- Исправление: ранний выход `if(!dlTime && !compareTo && dl === todayStr()) return 'today'`

**Итог тестирования:** 37 ✅, 0 ❌, 10 ⏭ (требуют браузер + OAuth)

**Создали документацию:**
- `test-scenarios.md` — 19 тестовых сценариев с критериями приёмки (приоритеты 🔴/🟡/🟢)
- `test-report.md` — полный отчёт о тестировании с таблицей результатов

### Решения

- `todayStr()` → используем `new Date()` с `.getFullYear()/.getMonth()/.getDate()` каждый вызов (не кешируем объект Date, т.к. страница живёт несколько часов)
- `run_tests.js` содержит собственную копию `dateStatus()` (извлечённую из HTML) — после исправления в dashboard.html нужно было обновить и её
- Запуск тестов: `TZ=Europe/Moscow node run_tests.js` — мок Date через `Date = class extends OriginalDate { constructor(...a) { super(a.length ? ...a : '2026-06-14T01:30:00+03:00') } }`

## Сессия 1 — Запуск и первые ошибки (2026-06-11)

### Что строили
Основную канбан-доску: 4 колонки, задачи, направления, архив, уведомления.

### Ошибки и решения

**404 при открытии через localhost:**
- Причина: сервер запускался не из папки с файлами
- Решение: `cd ~/Desktop/kanban && python3 -m http.server 8080`
- Долгосрочное решение: `launch.command` с `DIR="$(cd "$(dirname "$0")" && pwd)"`

**Импорт данных не работал (версия 1):**
- Причина: после `localStorage.setItem` делался `location.reload()`, но страница не успевала подхватить новые данные
- Решение: убрать reload, применять данные прямо в памяти: `tasks = data.tasks; AREAS = data.areas; render()`

**Импорт не работал (версия 2):**
- Причина: динамически созданный `input.click()` блокируется браузером
- Решение: статический `<label>` + `<input type="file" onchange="handleImportFile(this)">` прямо в HTML — работает везде

**Данные пропали при переезде:**
- Причина: `file://` и `http://localhost:8080` — разные origin, у каждого своё localStorage
- Решение: экспорт из `file://`, импорт в `http://localhost:8080`

## Сессия 2 — Google Calendar (2026-06-11)

### Что строили
Двустороннюю синхронизацию с Google Calendar через OAuth 2.0.

### Ключевые технические решения

**Способ связи задача ↔ событие:**
- Через `extendedProperties.private.taskId` — скрытое поле события, видно только нашему приложению

**Не создаём отдельный календарь:**
- Используем существующий выбранный пользователем — проще и меньше мусора

**Повторяющиеся события (Вариант А):**
- `singleEvents=true` — каждое вхождение разворачивается в отдельную задачу
- Прошедшие остаются как просроченные задачи, пользователь сам решает что с ними делать

**Окно синхронизации:**
- Текущая ISO-неделя + 32 дня вперёд — достаточно для охвата повторяющихся событий

**Разрешение конфликтов:**
- Last-write-wins по полю `lastModified` (Unix timestamp в миллисекундах)
- `lastModified` обновляется при любом изменении задачи (редактирование, перемещение, drag&drop, sync)

**Токен OAuth:**
- Хранится только в памяти (`let gcalToken = null`), не в localStorage
- Живёт ~1 час, после истечения запрашивается новый через попап

### Ошибки и решения

**OAuth зависал навсегда при нажатии «Подключить»:**
- Причина: браузер мог заблокировать попап, никакого таймаута не было
- Решение: таймаут 90 секунд с понятным сообщением об ошибке

**Ошибка 403: access_denied:**
- Причина: пользователь не добавлен в Test Users в Google Cloud Console
- Решение: APIs & Services → OAuth consent screen → Test users → добавить email

**Safari блокировал попапы:**
- Решение: Safari → Настройки → Сайты → Всплывающие окна → localhost → Разрешить

**Автосинк не работал при открытии:**
- Добавлено в INIT: `if(gcalConfig.clientId && gcalConfig.calendarId) syncFull();`

## Сессия 3 — Запуск и дистрибуция (2026-06-11)

### Что строили
Простой способ запуска и переноса на другой компьютер.

### Решение: launch.command
```bash
#!/bin/bash
DIR="$(cd "$(dirname "$0")" && pwd)"
cd "$DIR"
lsof -ti:$PORT | xargs kill -9 2>/dev/null
python3 -m http.server 8080 &>/dev/null &
trap "kill $SERVER_PID 2>/dev/null" EXIT
open "http://localhost:8080/dashboard.html"
wait $SERVER_PID
```

Двойной клик → сервер запустился → браузер открылся. При закрытии окна — сервер останавливается.

**Ключевой нюанс:** `DIR="$(cd "$(dirname "$0")" && pwd)"` — сервер всегда запускается из папки скрипта, независимо от того, из какого места был запущен.

### Стратегия дистрибуции
- **Для себя:** полная установка с Google Calendar через `launch.command`
- **Для дочерей:** базовая установка — двойной клик на `dashboard.html`, никакого терминала
- **Перенос данных:** экспорт JSON на старом компьютере → импорт на новом

## Сессия 6 — Исправление GCal синхронизации (2026-06-13)

### Проблема

После предыдущих исправлений «старые» задачи (изменённые до установки патча) по-прежнему не обновлялись в Calendar даже при ручном пересохранении и повторной синхронизации.

**Симптомы:**
- «Ангара» — в дашборде дедлайн 15 июня, в Calendar по-прежнему 11 июня
- «Написать физиотерапевту» — в дашборде дедлайн 19 июня и новое название, в Calendar 11 июня и старое название

### Корневая причина

1. `gcalEnsureToken()` не возвращает булево значение (всегда `undefined`)
2. Функция `forcePushAll()` проверяла `if(!ok)` где `ok = undefined` → всегда срабатывала как «нет токена»
3. Для ряда задач `gcalEventId` в localStorage указывал на события с несовпадающим ID → PATCH возвращал 404 → события создавались заново через POST → появлялись дубли

### Что сделано

**Исправления кода:**
- Добавлена функция `forcePushAll()` — принудительный push всех задач с дедлайном в Calendar без проверки временны́х меток
- Кнопка «⚡ Принудительно обновить» в модалке настроек GCal
- Исправлена проверка токена: `if(!gcalToken)` вместо `if(!ok)`
- Добавлены `console.log` в `syncFull()` для диагностики timestamp-ов

**Разовая очистка:**
- После `forcePushAll` появились дубли в Calendar (новые на правильных датах + старые на 11 июня)
- Пользователь вручную удалил старые события из Google Calendar

### Итог

Синхронизация работает корректно. Изменения дедлайна и других полей сохраняются при следующей синхронизации. Выпущена версия **v1.1**.
